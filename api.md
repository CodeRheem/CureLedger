# CureLedger Backend API Inventory

This document was generated by checking mounted Express routes in the backend code and comparing them with the required spec in backend.md.

## Base Path

All mounted backend endpoints are under:

- `/api/v1`

Health route (outside versioned API):

- `GET /health`

## Available Endpoints (Mounted and Reachable)

### Auth

- `POST /api/v1/auth/register-recipient`
- `POST /api/v1/auth/register-hospital`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/send-verification-email`
- `POST /api/v1/auth/verify-email`
- `GET /api/v1/auth/me` (protected)

### Campaigns

- `GET /api/v1/campaigns`
- `GET /api/v1/campaigns/pending` (protected: hospital/admin)
- `GET /api/v1/campaigns/:id`
- `POST /api/v1/campaigns` (protected: recipient)
- `PATCH /api/v1/campaigns/:id` (protected)
- `GET /api/v1/campaigns/:id/donations` (protected)

### Verification Workflow

Mounted via `router.use('/campaigns', verifyRoutes)`:

- `POST /api/v1/campaigns/:campaignId/verify` (protected: hospital)
- `POST /api/v1/campaigns/:campaignId/approve` (protected: admin)
- `POST /api/v1/campaigns/:campaignId/reject` (protected: hospital/admin)

### Donations

- `POST /api/v1/donate`
- `POST /api/v1/donate/webhook`

### Withdrawals

- `POST /api/v1/withdraw` (protected: recipient)
- `POST /api/v1/withdraw/:id/approve` (protected: admin)

### Admin

- `GET /api/v1/admin/recipients` (protected: admin)
- `GET /api/v1/admin/hospitals` (protected: admin)
- `POST /api/v1/admin/hospitals/:id/verify` (protected: admin)
- `GET /api/v1/admin/withdrawals/pending` (protected: admin)

### Payments

- `POST /api/v1/payments/initiate`
- `GET /api/v1/payments/:reference`
- `POST /api/v1/payments/webhook`

## Endpoints Still Needed (Compared to backend.md)

## 1. Missing Endpoints

- `POST /api/v1/campaigns/:id/documents`
  - Required for campaign document upload and AI analysis response.

- `POST /api/v1/admin/funds/divert`
  - Required admin function for redirecting funds between campaigns.

## 2. Path Mismatches (Implemented but different from backend.md)

- Spec: `POST /api/v1/auth/register`
  - Current: split into `POST /api/v1/auth/register-recipient` and `POST /api/v1/auth/register-hospital`

- Spec: `POST /api/v1/campaigns/:id/withdraw`
  - Current: `POST /api/v1/withdraw` with `campaignId` in body

- Spec: `POST /api/v1/admin/withdrawals/:id/approve`
  - Current: `POST /api/v1/withdraw/:id/approve`

## 3. Behavior Gaps vs Spec Notes

- `GET /api/v1/campaigns` currently lists approved campaigns but does not expose all listed query options from spec (`status`, `sortBy`, `sortOrder`).
- Webhook event dispatch endpoints/logic for:
  - `campaign.status_changed`
  - `document.ai_analyzed`
  - `donation.received`
  - `withdrawal.approved`
  are not implemented as explicit outbound webhook dispatching features.
